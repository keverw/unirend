## About

The `clientInfo` plugin normalizes basic client information (IP address, user agent), provides a correlation ID for tracing requests across services, and exposes response headers for request/correlation IDs. It decorates Fastify's `request` with a read‑only `clientInfo` object, and can trust client details forwarded from your SSR server to API endpoint (when not short-circuiting within the same server process).

> **The request ID itself is generated by the server, not this plugin.** `request.requestID` (a ULID by default, used for the envelope `request_id`) is set by the framework before any plugins run — see the [`getRequestID`](../ssr.md#shared-server-configuration) server option. `clientInfo` reads that value into `clientInfo.requestID` and derives the correlation ID from it. It is a different value from the access log's `reqID` (Fastify's native incremental `request.id`); see [api-envelope-structure.md](../api-envelope-structure.md#request-id-handling).

Use it to:

- Trace a user action across SSR → API hops using a correlation ID
- Normalize IP/User‑Agent, optionally honoring forwarded headers from trusted sources
- Emit `X-Request-ID` / `X-Correlation-ID` response headers for client-side tracking

## Features

- **Correlation ID**: Uses `X-Correlation-ID` when trusted/valid, falls back to the server's request ID
- **SSR‑aware**: Optionally trusts forwarded client details from private/allowed sources
- **Request decoration**: Adds a frozen `request.clientInfo` (and reflects the server's `request.requestID` into it)
- **Response headers**: Sends `X-Request-ID` and `X-Correlation-ID` by default
- **Configurable logging hooks**: Opt-in request and forwarding logs, or pass `logging: true` to enable all

## Usage

```typescript
import { clientInfo } from 'unirend/plugins';

const server = serveSSRBuilt(buildDir, {
  plugins: [clientInfo()],
});
```

Inside handlers/hooks you can access:

```ts
request.requestID; // string
request.clientInfo; // { requestID, correlationID, isFromSSRServerAPICall, IPAddress, userAgent, isIPFromHeader, isUserAgentFromHeader }
```

## Configuration

- `forwardedRequestIDValidator?: (id: string) => boolean`, Validator for the forwarded `X-Correlation-ID` from trusted SSR requests (default: ULID validation). Forwarded values that fail validation are ignored and the request ID is used as the correlation ID instead. (To customize the request ID itself, use the server's `getRequestID` option, not this plugin.)
- `setResponseHeaders?: boolean`, Send `X-Request-ID` and `X-Correlation-ID` (default: `true`)
- `trustForwardedHeaders?: (request: FastifyRequest) => boolean`, Callback that determines whether to accept forwarded client-info headers (default: returns `true` when `request.clientIP` is private. Otherwise forwarded headers are ignored and direct request values are used)
- `logging?: boolean | { requestReceived?: boolean; forwardedClientInfo?: boolean; rejectedForwardedHeaders?: boolean }` ,
  - `true`: enable all logs (request received, forwarded client info, rejected forwarded headers)
  - `false` or `undefined`: disable all logs (default)

> **These logs are diagnostic, not a request log.** They report what the plugin itself decides — the request it received and whether forwarded `X-SSR-*` headers were trusted or rejected — and are off by default. For normal per-request logging (method, URL, status code, response time), use the server's [`accessLog`](../ssr.md#access-logging) option instead. The most useful flags here are `forwardedClientInfo` and `rejectedForwardedHeaders` when debugging why forwarded client info isn't being trusted across an SSR → API hop; `requestReceived` overlaps with `accessLog`'s `events: 'start'`.

Client info shape set on `request.clientInfo`:

```ts
type ClientInfo = {
  requestID: string;
  correlationID: string | null;
  isFromSSRServerAPICall: boolean;
  IPAddress: string;
  userAgent: string;
  isIPFromHeader: boolean;
  isUserAgentFromHeader: boolean;
};
```

## Examples

```typescript
// Enable request logging
clientInfo({
  logging: { requestReceived: true },
});

// Only trust forwarded headers behind an allowlist/proxy check
clientInfo({
  trustForwardedHeaders: (request) =>
    request.clientIP?.startsWith('10.') === true,
});

// Disable response headers if you prefer not to expose IDs
clientInfo({ setResponseHeaders: false });

// Enable all logging with a single flag
clientInfo({ logging: true });

// Explicitly disable all logging
clientInfo({ logging: false });
```

## Forwarded Headers (SSR)

When `trustForwardedHeaders` returns `true`, the plugin will honor these request headers if present:

- `X-SSR-Request: "true"`, Marks the request as originating from SSR (`isFromSSRServerAPICall: true`)
- `X-SSR-Original-IP: <client-ip>`, Source IP from the browser/client
- `X-SSR-Forwarded-User-Agent: <ua>`, Original client user agent
- `X-Correlation-ID: <id>`, Correlation ID for tracing (validated via `forwardedRequestIDValidator`)

This lets your SSR server forward the user's client details to your API so both hops share the same correlation ID. See also: `docs/ssr.md`.

### Deployment Note

In production behind reverse proxies or load balancers, configure Fastify's `trustProxy` or `getClientIP` so `request.clientIP` reflects the real client IP. Unless you provide a custom `trustForwardedHeaders`, the plugin only accepts forwarded client-info headers when `request.clientIP` is private. Without one of those in place, `request.clientIP` may not reflect your proxy chain correctly.

- On SSR and API servers, set `fastifyOptions.trustProxy` or `getClientIP` when appropriate. Common `trustProxy` options are `true`, a trusted IP/CIDR value like `'127.0.0.1'` or `'127.0.0.1,192.168.1.1/24'`, a trusted IP/CIDR list like `['127.0.0.1', '10.0.0.0/8']`, or a custom trust function with signature `(address: string, hop: number) => boolean`. Fastify also supports numeric hop counts.
- Example: `serveSSRBuilt(buildDir, { fastifyOptions: { trustProxy: true }, plugins: [clientInfo()] })`
- Example with `getClientIP`:

```ts
serveSSRBuilt(buildDir, {
  getClientIP: (req) => {
    // Pseudo-code: only trust the external reverse proxy header when the request came from
    // a proxy or load balancer range you control.
    const fromTrustedProxyRange = isTrustedProxyRange(req.ip);
    const cfIP = req.headers['cf-connecting-ip'];

    if (fromTrustedProxyRange && typeof cfIP === 'string' && cfIP) {
      return cfIP;
    }

    return req.ip;
  },
  plugins: [clientInfo()],
});
```

- You can also use `trustProxy` and `getClientIP` together. For example, `trustProxy` can help you trust your load balancer and `getClientIP` can read the original client IP from an external reverse proxy header such as `CF-Connecting-IP`.
- You can also provide `trustForwardedHeaders` to further restrict when clientInfo accepts `X-SSR-Request`, `X-SSR-Original-IP`, `X-SSR-Forwarded-User-Agent`, and `X-Correlation-ID`. This does not change `request.clientIP`. It only controls whether those forwarded headers are used.

```ts
clientInfo({
  // e.g., only accept forwarded details when calls originate from your SSR/LB network
  trustForwardedHeaders: (request) =>
    request.clientIP?.startsWith('10.') === true,
});
```

## Relationship to `request.clientIP`

`clientInfo` builds on top of the core `request.clientIP` — they are related but answer different questions:

- `request.clientIP` (always set by the framework) is the resolved IP of the **current connection** — the `getClientIP` result or `request.ip`. This is what the built-in access log records (`{{ip}}` / `ctx.ip`), and what you'd use for connection-level decisions like rate limiting.
- `request.clientInfo.IPAddress` (and `.userAgent`) is the **real end user** across an SSR → API hop: it starts from `clientIP`, but on a trusted SSR-forwarded request it is replaced with the original browser values from `X-SSR-Original-IP` / `X-SSR-Forwarded-User-Agent`. The `isIPFromHeader` / `isUserAgentFromHeader` flags tell you when that happened.

| Scenario                                | `request.clientIP`         | `request.clientInfo.IPAddress`             |
| --------------------------------------- | -------------------------- | ------------------------------------------ |
| Direct browser → server                 | the user's IP              | same value                                 |
| Browser → SSR → API (on the API server) | the **SSR server's** IP    | the **original browser** IP (forwarded)    |
| Always present?                         | yes (core)                 | only when `clientInfo` is registered       |

**Suggestion:** when `clientInfo` is registered and you write access logs to your own sink (DB, analytics), prefer `request.clientInfo.IPAddress` / `.userAgent` and `request.clientInfo.correlationID` over the connection-level `clientIP` / raw User-Agent — they attribute the record to the real end user and tie SSR → API hops together.

> **Timing:** `request.clientInfo` is populated by this plugin's `onRequest` hook, which runs **after** access logging is registered. So clientInfo fields are available in the `accessLog.onResponse` hook (and finish/response templates), but **not** in `accessLog.onRequest`, where `request.clientInfo` is still unset. Read clientInfo fields in `onResponse`. (`request.requestID` and `request.clientIP` are core and available in both hooks.)

## Response Headers

By default, the plugin adds these to every response (configurable via `setResponseHeaders`):

- `X-Request-ID`, Unique ID for this request
- `X-Correlation-ID`, Correlation ID for the overall action (defaults to the request ID if not provided/valid)
